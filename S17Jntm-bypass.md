# mYjJYFSJwydN.exe 破解方案（v2 — 基于实测修正）

> 更新: 2026-07-01 | 基于 ProcMon + Wireshark + 二进制补丁实测

---

## 一、修正后的真实架构

### 1.1 实际验证流程（已确认）

```
mYjJYFSJwydN.exe (VMP 3.x 保护)
  │
  ├── 启动时
  │     GET http://jinitaimei.qs115.com/ping.txt
  │     响应: 135字节 hex加密数据 ← 用于本地验证
  │
  ├── 用户点击登录
  │     ├── 输入「卡密」
  │     ├── 程序内部(VC++/易语言)处理卡密
  │     ├── 使用 ping.txt 数据进行本地校验
  │     ├── NO 网络请求（验证完全本地）
  │     └── 结果: 弹窗"卡密无效" 或 通过
  │
  └── CEFlib/（浏览器功能层，非验证层）
        ├── FBrowserVIP.dll → 浏览器指纹伪装 + CEF控制
        │   （不是验证DLL，验证结果不影响"卡密无效"）
        ├── FBrowserCEF3lib.dll
        └── libcef.dll
```

### 1.2 关键修正（vs v1 方案）

| 原假设 | 实测真相 |
|--------|---------|
| FBrowserVIP.dll = 核心验证 | ❌ 它是浏览器指纹伪装 DLL，**不是验证DLL** |
| SetKey → IsLicenceKey 调用链 | ❌ 验证在 VMP 内部完成，不经过 DLL |
| patch FBrowserVIP 可绕过验证 | ❌ patch 后程序启动正常但登录仍报"卡密无效" |
| 验证涉及服务端通信 | ⚠️ 点击登录后**无网络请求**，验证完全本地 |
| 可 DLL 劫持接管验证 | ❌ 自定义加载器，不走 LoadLibrary |

## 三、所有可行方案（按成功率排序）

### Tier 1：高成功率（3-6 小时）

#### 方案 A：ProcMon 深挖验证行为 ⭐⭐ 成功率: 60%

**原理：** 程序点"登录"后一定有某种 IO 行为——读写文件、注册表、进程创建等。找出它做了什么就能找到突破口。

**操作：**
1. 运行 ProcMon，过滤 `mYjJYFSJwydN.exe`
2. **清除所有已有的 DLL 加载事件**（菜单 Edit → Clear Display）
3. 点程序"登录"按钮
4. 观察新增事件中的 `CreateFile` / `RegCreateKey` / `WriteFile` / `ReadFile`
5. 重点看程序目录和 `%TEMP%` 目录的操作

#### 方案 B：Hosts 劫持 + 伪造 ping.txt 响应 ⭐ 成功率: 30%

**原理：** 如果验证依赖 ping.txt 内容，修改返回内容可能改变验证结果。

**操作：**
1. 修改 hosts 加入 `127.0.0.1 jinitaimei.qs115.com`
2. 管理员运行本地 HTTP 服务器返回空响应或假数据
3. 观察"卡密无效"是否变化

#### 方案 C：Cheat Engine — ping.txt 数据流追踪 ⭐⭐ 成功率: 50%

**原理：** ping.txt 的 135 字节 hex 数据必须在内存中参与验证，追踪这些字节的流向就能找到验证代码入口。

**操作：**
1. 附加 CE 到程序
2. 搜索 hex: `76 62 60 3D 7F 7E 75 3E 73 7F 7D 1D 1A 76 62 60`
3. 右键 → `Find out what accesses this address`
4. 点"登录"按钮触发
5. 捕获访问代码的指令地址（在 VMP 区域中）

---

### Tier 2：中等成功率（1-2 天）

#### 方案 D：Overlay 数据分析 ⭐⭐⭐ 成功率: 40%

**原理：** 主程序尾部 1.3MB overlay 数据未被 PE 解析，可能是加密配置、验证白名单。提取分析。

**操作：**
1. 用 `dd` 或 Python 提取 EXE 尾部 overlay: `dd if=mYjJYFSJwydN.exe of=overlay.bin bs=1 skip=20893696`
2. 分析 overlay 数据结构（熵值、文件签名、字符串）

#### 方案 E：Cheat Engine — 修改验证结果标志 ⭐⭐⭐ 成功率: 40%

**原理：** 运行时在内存中找"验证通过/失败"的标志位，改它。

**操作：**
1. 先点"登录"让状态设为"失败"
2. CE 搜索 0
3. 再点一次（或找其他方式重置状态）
4. CE 搜改变的值（差值法）

#### 方案 F：WinDivert 驱动层拦截 ⭐⭐⭐ 成功率: 35%

**原理：** 在驱动层拦截 jinitaimei.qs115.com 的所有流量并修改响应。

**操作：**
1. 管理员运行 WinDivert
2. 过滤规则: `outbound and tcp.DstPort == 80 and tcp.PayloadLength > 0`
3. 修改 ping.txt 响应内容

---

### Tier 3：低成功率但彻底（3-7 天）

#### 方案 G：x64dbg + ScyllaHide 硬件断点 ⭐⭐⭐⭐⭐ 成功率: 20%

**原理：** 硬件断点绕过 VMP 反调试，动态追踪验证逻辑。

**操作：**
1. 用 ScyllaHide 插件启动 x64dbg
2. 在 `MessageBoxA`/`MessageBoxW` 设断点
3. 触发"卡密无效"弹窗，捕获调用堆栈
4. 回溯到验证决策代码

#### 方案 H：逆向 ping.txt 加密算法 ⭐⭐⭐⭐ 成功率: 15%

**原理：** 分析 135 字节 hex 数据的加密结构，尝试解密。

**操作：**
1. 用 Crypto++ 的 AES/SHA 尝试解密
2. 测试已知密钥（MySecureKey_2025!@#）
3. 猜测明文格式（JSON/序列化/自定义结构）

## 五、方案总结

### 已排除的方案

| 方案 | 排除原因 |
|------|---------|
| FBrowserVIP.dll 劫持 | 主程序用自定义加载器非 LoadLibrary，且验证不在 DLL 中 |
| FBrowserVIP.dll patch | patch 后程序正常启动但登录仍"卡密无效" |
| ProcMon DLL 诊断 | 已完成，确认 FBrowserVIP 不走 LoadLibrary |
| Wireshark 抓包 | 已完成，确认 ping.txt + 登录无新请求 |
| 代理 DLL (#pragma 转发) | 自定义加载器不支持 forwarder |

### 当前最佳突破口

**ProcMon 深挖**——这是当前时间成本最低、信息量最大的选项。点"登录"触发验证后，看 ProcMon 里发生了什么，这是破解此程序最快、最直接的路径。
